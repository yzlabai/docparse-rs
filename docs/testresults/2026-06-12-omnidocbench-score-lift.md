# 测评 · OmniDocBench 提分实验(Phase 6 B1–B4)

> 2026-06-12 · 计划 [omnidocbench-score-lift.md](../plans/omnidocbench-score-lift.md)
> 数据/脚本同 [2026-06-12-omnidocbench.md](2026-06-12-omnidocbench.md)。本文记录"还能怎么提分"的系统性探查与**主要为负的结论**。

## 0. 一句话结论

学术表代理 Overall ≈75 的瓶颈是**学术表 TEDS 0.517**;本轮逐一试了所有便宜旋钮(loop-trim 救退化、area 下采样、抬 max_tokens、**抬输入分辨率**),**全部 no-op、反降、或净≈0 但破速度门**。看图核对:0.517 是 **UniRec-0.1B 在大/宽/密学术表上的真实天花板**——**固定 960×1408 输入分不清多行多列**(主因 UNDER-segment 大表 + NO-PRED 超宽表),over-split 列表格是少数。分辨率实验铁证(#11 0→0.966)证明杠杆真在分辨率,但 0.1B **对分辨率脆 + 3× 慢**,无法在保速下干净利用。落地两项**零回归**(B1 评测对称、B2 失控退化抢救);B3/分辨率/max_tokens/B4/B5 据证据**否决/缓办**。真杠杆=更大/多分辨率表模型 或 `--vlm-tables` 服务(均在内嵌+保速域外)。

## 3. 失败模式解剖(为什么便宜旋钮无效)

### 3a. 看图核对真值(纠正初判)
初判把 #4/#16 当"评测口径反噬(我们更忠实、被粗标真值扣)"。**把表区裁出来看图后纠正:恰恰相反——是模型 OVER-SPLIT 错误**:
- **#4**:5 列、真**2 数据行**,"Type"列每行是个项目符列表(`<ul><li>`)。GT 3×5(列表留在格内)**才对**;模型把每个 bullet 拆成独立行(6 行)、其余列留空 → 错。
- **#16**:4 列对照表,每列是**不等长的竖排清单**(行本不对齐)。GT 2×4(清单留格内)**是忠实表示**;模型硬塞 15 假行 → 错。

教训:口径判断必须**看图**,不能只对 GT/pred 的 HTML 形态猜。

### 3b. 逐表 pred-vs-GT 行列数诊断(30 表)
| 失败类 | 表(GT→pred) | 占比 |
|---|---|---|
| **UNDER-segment / 截断(大/宽表)** | #21 56→27 行、#20 55→35、#26 43→24、#19 27→14、#24 16→6、#11 7→1、#18 16→1 | **7** |
| **NO-PRED(超大表)** | #6 18×**50 列**→无、#22 21×11→无 | **2** |
| **OVER-SPLIT(列表格)** | #4 3→6、#16 2→15、#14 25→96 | **3** |

**主因是大/宽表 UNDER-segment + NO-PRED(9/低分表),非 over-split(3)**。#21 输出**完整**(有 `</table>`)却只 27 行 vs GT 56——不是 token 截断(抬 max_tokens 反降),是**模型在固定 960×1408 输入下分不清那么多细行/细列→并行或漏行**;#6 是 50 列宽表,缩到 960 宽每列 ~19px→读不出→无输出。**根因=模型固定输入分辨率 + 容量在大/密表上的天花板**。

### 3c. 分辨率实验(决定性,否决)
模型有动态空间维 → 试把输入上限 960×1408 抬到 **1280×1792**,学术 30 表:

| | 均值 | 速度 |
|---|---|---|
| 960×1408(native) | **0.517** | ~9s/表 |
| 1280×1792 | **0.527**(+0.010,噪声内) | **~26s/表(≈3×)** |

逐表**剧烈分裂**:救资源饥饿表(#11 **0→0.966**、#24 0.369→0.612、#18 0→0.170、#13 +0.104、#4 +0.065),但**扭曲本已合身的表**(#10 0.413→**0.000**、#15 0.618→0.162、#30 0.972→0.855、#17 −0.097)。
- **结论**:分辨率是**真杠杆**(#11 0→0.966 铁证),但 0.1B 模型**对分辨率脆**——训练在 ≤960×1408,偏离 native 救饥饿表却扭曲合身表,**净 ≈0 且 3× 慢**(破 ≤5s/表 spike 门)。**否决 blanket bump**;自适应(仅大表升分辨率 + 二选优)要路由信号+二跑+挑优启发,复杂度高、上限被模型脆性封顶,缓办。

`<br>`/`<ul>` 计数与分数**不相关**(粗标表均值 0.587 反高于干净表 0.471)。综上:学术表瓶颈 = **0.1B 模型固定分辨率+容量在大/宽/密表上的真实天花板**;分辨率/解码侧旋钮均无法在保速下干净利用。

## 4. 落地(零回归)

### B1 · 评测 strip_math 对称(评测侧)
端到端表评测 pred 侧 `ir_table_cells` 此前只过 `_norm`、GT 侧过 `strip_math` → 含行内数学的格(`\(d_w\)` vs `d_w`)假性失配。修为对称。**评测正确性修复**,非提分手段。

### B2 · EOS 门控失控退化抢救(unirec.rs)
判据:**退化=跑满 token 上限且不吐 EOS 的重复环;好表自带 EOS 自终止**。只在"撞上限且无 EOS"时找重复尾块裁掉、保留好前缀;自终止的表(含整列相同值的合法表)**一律原样信任**。
- 学术 30 表**逐字不变**(0.517),零回归——确认安全;
- 价值在**域外 runaway**(评测集 CJK=韩文,refine 路径此前被 `looks_degenerate` 整表丢弃)→ 抢救前缀而非全丢;**域外全集收益候夜跑验证**;
- 首版激进 in-loop 早停的好表误触发已被 EOS 门控消除(教训:安全特性验收语料必须含"含重复内容的合法表")。
- 单测:`loop_trim_rescues_degenerate_tail`(退化尾被裁、合法差异表不裁、不足次数不裁)。

## 5. 否决/缓办(据证据)

- **B3 area 下采样**:tables 近中性(多数下采样 <2×,area≈bilinear);且它改的是 UniRec **公共预处理**(表/公式/转写三路),**未在公式/转写上验证零回归** → **缓办**,定档"正确的滤波器,需三任务齐验后再上"。
- **B4 小 beam**:失败解剖显示主因是**大表 UNDER-segment / 超宽表 NO-PRED**(分辨率/容量),beam 只救边际 token 决策,对"分不清行列"**预期低收益**;且全部其它旋钮均无效/反向 → **不投**,理由记录(避免又一轮"写进去再回退")。
- **抬输入分辨率**:净 ≈0(救饥饿表 = 扭曲合身表)+ 3× 慢破速度门 → 否决 blanket;自适应升分辨率复杂度高、上限被模型脆性封顶 → 缓办(§3c)。
- **max_tokens 抬高**:实测反而降(§2)→ **不做**,2000 维持。
- **B5 多表召回**:学术单模块就 0.517 封顶,检测层非主瓶颈 → 候真官方 harness。

## 6. 真正的提分杠杆(均在"便宜/保速"域外)

1. **更大 / 多分辨率表模型**:UniRec/OpenDoc 仅 0.1B、固定 ≤960×1408 输入;大/宽/密表分不清行列是它的硬限,更大或可变分辨率才解,与"内嵌单二进制 + 保速"身份冲突;
2. **VLM 服务路径**(`--vlm-tables` 接 Qwen2.5-VL 级):难表交外接服务——**已存在**,只是非内嵌默认档;
3. **(已否)口径**:看图核对后 #4/#16 是模型 over-split 错误**而非** GT 粗标——口径不是主因。

## 7. 沉淀教训

1. **分数可疑先怀疑评测,但要看图坐实**(lesson #1):e2e 漏 strip_math 是真不对称;但 #4/#16 初判"口径反噬"**看图后被推翻**(实为模型 over-split)——口径判断不能只对 HTML 形态猜。
2. **抬"显然该有用"的旋钮也要量**:分辨率确实是真杠杆(#11 0→0.966),但 0.1B 对它脆,净 ≈0 且破速度门——杠杆真≠能用。
3. **安全特性要先看"含对抗内容的合法样例"**(B2 首版):好表里整列相同值就能假触发退化检测。
4. **负结论建到能下定论**:逐旋钮(loop-trim/area/max_tokens/分辨率)+ 看图失败解剖 + 行列数诊断,才敢说"便宜/保速域无解,瓶颈是 0.1B 模型本身",而非"还没试对"。
